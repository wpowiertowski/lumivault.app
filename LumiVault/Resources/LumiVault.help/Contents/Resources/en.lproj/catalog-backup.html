Catalog Backup & Restore

The catalog (catalog.json) is the central database of your LumiVault library. It records every album, photo, hash, and storage location. LumiVault automatically distributes backup copies to keep it safe.

Automatic Backups

After each import, LumiVault saves a copy of catalog.json to:

  • Each external volume — stored at the root of the LumiVault folder on each configured drive.
  • Backblaze B2 — uploaded alongside your photos (if B2 is enabled).

Restoring from Backup

If your local catalog is lost or corrupted, you can restore from any backup copy:

  • From a file — click Restore from File on the welcome screen or in Settings > General. Select a catalog.json file from any location.
  • From a volume — click Restore from Volume and select an external drive. LumiVault finds the catalog on that volume automatically.
  • From B2 — click Restore from B2. LumiVault downloads the latest catalog from your configured B2 bucket.

Detect Existing Catalog

The Detect Existing button scans connected volumes for a catalog.json and offers to import it. Useful when setting up LumiVault on a new Mac with your existing archive drives.

Important: Restoring a catalog replaces your current catalog entirely. Make sure you're restoring from the most recent backup to avoid losing recent imports.

Recovering from Data Loss

If LumiVault has lost its local state (fresh install, deleted container, or empty library), follow this order to recover. The order matters because volumes can only re-link to existing catalog entries.

  1. Restore the catalog first — from a volume, a file copy, or B2. Your albums and images appear in the sidebar.
  2. Re-enter B2 credentials in Settings > B2 if you use B2 backup. Credentials are stored locally and are not part of the catalog.
  3. Re-toggle iCloud Sync in Settings > iCloud if you use it. The toggle state is local.
  4. Add each external volume in Settings > Volumes. After each, run the Sync to Volume sheet that opens. It detects existing files on the drive by hash and re-links them to the catalog without copying.
  5. Run Integrity Check from Settings > Integrity to verify hashes and PAR2 coverage across all volumes.

What the catalog carries — and what it doesn't

The catalog is the source of truth for what you've archived. Some state lives outside it and must be rebuilt or re-entered after a restore:

StateIn the catalog?After restore
Album list, image filenames & hashesYesRestored
PAR2 sidecar filenamesYesRestored
B2 file IDs & encryption metadataYesRestored
Per-volume file locationsNoEmpty — rebuilt by Sync to Volume (step 4)
External volume bookmarks & IDsNoRe-add via Settings > Volumes
B2 credentialsNoRe-enter in Settings > B2
iCloud / B2 sync togglesNoRe-enable in Settings
ThumbnailsNoRegenerate on demand as you browse
Perceptual hashes (near-duplicate detection)NoRebuilt by Near Duplicates view or re-import
Link to original Photos albumNoRestored on re-import of that album
Note on volume IDs: Each external volume gets a fresh identifier when you add it back — the pre-restore ID is not stored on the drive itself. This is harmless: the Sync to Volume sheet matches files by SHA-256 hash, not by volume ID, so existing archives are re-linked correctly.
Don't add volumes before restoring. If you add a volume while the catalog is empty, the Sync to Volume sheet will report “0 images in catalog” and have nothing to link against. Restore first.
